# Database — Sprint 2

This document is the canonical reference for the Opportunity Hunter
database foundation. The schema is defined by `docs/DATABASE_DESIGN.md`
in the project root; this file documents the *implementation* of that
schema in PostgreSQL / Supabase.

## Stack

- **Supabase** (managed PostgreSQL 15+).
- **TypeScript** with strict mode (see `tsconfig.json`).
- **PostgREST** via `@supabase/supabase-js` and `@supabase/ssr` for
  typed queries.
- **Migrations** are plain `.sql` files under `supabase/migrations/`.
- **Repository layer** lives in `src/lib/db/repositories/`.

## High-level flow

```
                ┌───────────┐
                │  sources  │  external platforms (Reddit, App Reviews, …)
                └─────┬─────┘
                      │ "source" is a string, not a FK
                      │  (see DATABASE_DESIGN.md)
                      ▼
              ┌───────────────┐
              │   raw_posts   │  harvested content
              └───────┬───────┘
                      │ 1
                      ▼ N
            ┌───────────────────┐
            │   pain_points     │  extracted pains
            └─────────┬─────────┘
                      │ (joined in Sprint 6+)
                      ▼
           ┌────────────────────┐
           │   pain_clusters    │  themes
           └─────────┬──────────┘
                     │ 1
                     ▼ N
            ┌────────────────────┐
            │   opportunities    │  scored themes
            └─────────┬──────────┘
                      │ 1
                      ▼ N
            ┌────────────────────┐
            │   startup_ideas    │  product briefs
            └────────────────────┘
```

## Tables

| Table | Purpose | Notable columns |
| --- | --- | --- |
| `sources` | External platforms we collect from. | `name`, `type`, `url` |
| `raw_posts` | Posts harvested from a source. | `source` is a string, not a FK |
| `pain_points` | Pains extracted from a post. | `severity`, `buying_intent` in `[0, 1]` |
| `pain_clusters` | Themed groups of pains. | `cluster_name`, `description` |
| `opportunities` | Quantified score for a cluster. | `score` in `[0, 100]` |
| `startup_ideas` | Concrete product brief from an opportunity. | `problem`, `solution`, `mvp`, `pricing` |

Every primary key is a `uuid` generated by `gen_random_uuid()` (provided
by the `pgcrypto` extension enabled in
`supabase/migrations/00000000000001_extensions.sql`).

## Foreign keys

| From | → To | On delete |
| --- | --- | --- |
| `pain_points.raw_post_id` | `raw_posts.id` | `cascade` |
| `opportunities.cluster_id` | `pain_clusters.id` | `cascade` |
| `startup_ideas.opportunity_id` | `opportunities.id` | `cascade` |

`raw_posts.source` is **not** a FK by design (per
`DATABASE_DESIGN.md`). The application layer is responsible for keeping
`source` in sync with `sources.name`.

`pain_points → pain_clusters` (many-to-many) is **not** modelled yet.
Sprint 6+ will introduce a join table once the clustering layer lands.

## Indexes

Performance-critical indexes (full list lives in the migration files):

- `sources`: `type`, `created_at desc`.
- `raw_posts`: `source`, `score desc`, `created_at desc`, plus a
  composite `(source, score desc, created_at desc)` for "top posts
  per source" queries.
- `pain_points`: `raw_post_id`, `category`, `severity desc`,
  `buying_intent desc`, `created_at desc`.
- `opportunities`: `cluster_id`, `score desc`, `frequency desc`,
  `buying_intent desc`, plus `(cluster_id, score desc)`.
- `startup_ideas`: `opportunity_id`.

## Constraints

Each table has:

- A `check` constraint on every `text` column to bound its length and
  guard against empty strings.
- A `check` constraint on every `numeric(4,3)` to enforce `[0, 1]`.
- A `check` constraint on `opportunities.score` to enforce `[0, 100]`.
- A `check` constraint on `raw_posts.score` to keep it non-negative.

The spec deliberately omits `created_at` from `pain_clusters`,
`opportunities`, and `startup_ideas`, so we don't add them.

## Row Level Security

RLS is enabled on every table. Sprint 2 ships with a single permissive
`service_role` policy per table (full access). Once auth is wired up in
later sprints we will tighten the policies to per-user / per-org access
without changing the schema.

## Types and the repository layer

- The hand-written TypeScript schema lives in
  `src/types/database.types.ts`. It mirrors the SQL one-for-one and
  exports `Row`, `Insert`, `Update`, and the `Database` container type
  that `createClient<Database>(...)` is generic over.
- The `Uuid`, `Decimal3`, `Decimal6`, and `Json` aliases are exported
  alongside the table types. Numeric columns are exposed as `string`
  to avoid float drift when round-tripping through JSON.

Repositories (one per table) live under
`src/lib/db/repositories/<table>.repository.ts` and follow a single
contract:

```ts
const repo = await SourcesRepository.create();
const source = await repo.findById(id);
```

Every repository:

- Accepts a `SupabaseClient<Database>` via its constructor (so tests
  can inject a mock).
- Throws typed `RepositoryError` subclasses (`NotFoundError`,
  `UniqueViolationError`, `ForeignKeyViolationError`,
  `CheckViolationError`) instead of raw `PostgrestError`.

## Environment

| Variable | Where | Required | Notes |
| --- | --- | --- | --- |
| `NEXT_PUBLIC_SUPABASE_URL` | client + server | yes | Project URL from Supabase. |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | client + server | yes | Anon key — subject to RLS. |
| `SUPABASE_SERVICE_ROLE_KEY` | server only | server-side jobs | Bypasses RLS. |
| `SUPABASE_DB_URL` | tooling | optional | Direct Postgres URL for the Supabase CLI / psql. |

A copy-paste template lives in `.env.example`. Real values belong in
`.env.local` (git-ignored).

## What is explicitly **out of scope** for Sprint 2

- The Reddit collector (Sprint 4).
- Any AI / LLM integration (Sprint 5+).
- Many-to-many `pain_points ↔ pain_clusters` mapping (Sprint 6+).
- Auth (Sprint 10+).
